### 1.2 备选方案对比
| 方案 | 优点 | 缺点 | 未选原因 |
|------|------|------|---------|
| Node.js + Express | 前后端统一、开发快 | 生态成熟度、金融场景稳定性弱 | 金融场景需要企业级稳定性 |
| Go + Gin | 性能高、并发强 | 生态不如Java成熟、团队技术栈 | 团队Java基础更好 |
| Python + FastAPI | 与AI服务统一 | 性能瓶颈、GIL限制 | 仅用于AI服务，主业务用Java |
| Spring Cloud | 微服务全家桶 | 运维复杂、MVP过度设计 | 先单体，后续拆分 |

## 5. 关键技术点

### 5.1 实时行情推送（WebSocket + STOMP）
- **问题**：行情数据需要实时推送给前端
- **方案**：
  - Spring Boot集成Spring WebSocket + STOMP协议
  - 后端定时任务（XXL-JOB每3秒）从聚宽API拉取行情 → 存入Redis → 发布到WebSocket Topic
  - 前端订阅 `/topic/stock/{symbol}` 接收实时价格
  - 使用Redis Pub/Sub做集群间消息广播
- **备选**：SSE（Server-Sent Events）更简单，但WebSocket支持双向通信

### 5.2 交易并发处理（分布式锁 + 数据库事务）
- **问题**：高并发下防止超卖、重复下单、资金扣减不一致
- **方案**：
  - **Redis分布式锁**：`SET stock:lock:{userId}:{symbol} NX EX 10` 防止同一用户同一股票并发下单
  - **数据库行级锁**：`SELECT * FROM user WHERE id = ? FOR UPDATE` 锁定用户资金
  - **乐观锁**：用户表增加 `version` 字段，防止资金并发修改
  - **幂等性校验**：订单号 `orderNo` 唯一索引，防止重复提交
  - **数据库事务**：`@Transactional` 保证原子性
  - **异步处理**：下单后发送消息到RabbitMQ，消费者异步处理成交

```java
@Service
public class TradeServiceImpl implements TradeService {

    @Transactional(rollbackFor = Exception.class)
    public OrderResponse createOrder(Long userId, OrderRequest request) {
        // 1. 获取分布式锁
        String lockKey = "order:lock:" + userId + ":" + request.getSymbol();
        boolean locked = redisLock.tryLock(lockKey, 10, TimeUnit.SECONDS);
        if (!locked) {
            throw new BusinessException("操作过于频繁，请稍后重试");
        }

        try {
            // 2. 校验交易规则（时间、涨跌停、资金）
            validateTradeRules(userId, request);

            // 3. 锁定用户资金（行级锁）
            User user = userMapper.selectByIdForUpdate(userId);

            // 4. 扣减资金/增加持仓
            if (request.getType() == OrderType.BUY) {
                user.setVirtualBalance(user.getVirtualBalance().subtract(totalAmount));
            }

            // 5. 保存订单
            Transaction transaction = new Transaction();
            transaction.setOrderNo(generateOrderNo());
            transaction.setStatus(OrderStatus.PENDING);
            transactionMapper.insert(transaction);

            // 6. 发送异步消息处理成交
            rabbitTemplate.convertAndSend("trade.exchange", "trade.routing", transaction);

            return new OrderResponse(transaction.getOrderNo());
        } finally {
            redisLock.unlock(lockKey);
        }
    }
}
```

### 5.4 策略回测性能优化
- **问题**：历史数据量大（单只股票10年日线约2500条），回测计算慢
- **方案**：
  - **数据预聚合**：按周/月预计算均线、MACD等指标，存入Redis
  - **并行计算**：Java 21虚拟线程 `ExecutorService.newVirtualThreadPerTaskExecutor()` 并行处理多只股票
  - **结果缓存**：相同参数（策略+股票+时间范围）直接返回缓存
  - **异步执行**：回测任务提交到线程池，前端轮询进度

## 7. 安全设计
- **认证授权**：Spring Security + JWT双令牌（Access 15分钟 + Refresh 7天）
- **密码加密**：bcrypt哈希，cost factor 12
- **API限流**：RateLimiter令牌桶，每用户每分钟100次
- **SQL注入**：MyBatis-Plus参数化查询，禁止字符串拼接SQL
- **XSS防护**：前端转义输出，后端Content-Security-Policy响应头
- **CSRF**：SameSite=Strict Cookie + CSRF Token（非JWT接口）
- **敏感操作**：交易确认需二次密码验证（交易密码）
- **日志脱敏**：身份证号、银行卡号等敏感字段打码存储
- **接口签名**：关键接口（交易）增加请求签名防篡改
